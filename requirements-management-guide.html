Workspace Structure

Your Notion workspace is organized as a top-level program space containing six linked databases plus a set of static reference pages. Everything lives under a single parent page so permissions, exports, and search all scope correctly.

Notion Sidebar Hierarchy
📁 [PROGRAM NAME] ← top-level parent page │ ├── ⚡ Program Dashboard ← linked views of all databases │ ├── 📊 Databases ← all 6 databases live here (linked elsewhere) │ ├── T1 — Mission Requirements │ ├── T2 — System Requirements │ ├── T3 — Subsystem Requirements │ ├── T4 — Implementation Requirements │ ├── Change Requests │ └── V&V Records │ ├── 📋 Design Reviews │ ├── SRR — [Date] │ ├── PDR — [Date] │ ├── CDR — [Date] │ └── [template] │ ├── 📖 Reference │ ├── Abbreviations & Definitions │ ├── Verification Methods Guide │ ├── Req Writing Style Guide │ └── Tier Ownership Map │ └── 🗂️ Archive └── Superseded Requirements
Key Principle

Never duplicate database entries. Create linked views (Notion: "Link to database") to show filtered subsets of a database on different pages. A requirement lives in exactly one place; every other appearance is a view.

Setting Up Two-Way Relations in Notion

Notion relations are bidirectional. When you create a relation from T2 → T1, Notion automatically creates the reverse property in T1. Name both sides clearly:

Relation Setup Checklist
// T2 database Property: "Parent T1" → Relation to T1 DB // Notion auto-creates in T1: Property: "T2 Children" ← rename this immediately // T3 database Property: "Parent T2" → Relation to T2 DB // Rename reverse in T2 to: Property: "T3 Children" // T3 ↔ V&V (bidirectional many-to-many) Property: "V&V Records" → Relation to V&V DB // Rename reverse in V&V to: Property: "Requirements Verified" // T2 ↔ Change Requests Property: "Change Requests" → Relation to CR DB // Rename reverse in CRs to: Property: "Affected Requirements"

Requirement Lifecycle

Every requirement moves through a defined status lifecycle. The transitions have explicit triggers and owners — this prevents requirements from getting stuck in ambiguous "in progress" states.

DRAFT
Being written. May be incomplete, untraceable, or not yet verifiable.
ACTIVE
Complete, traced, verifiable. Design team is designing to it. Not yet frozen.
BASELINED
Frozen at a design review gate. Changes require a formal Change Request.
IN CHANGE
A Change Request is open against this requirement. Implementation paused pending decision.
IN VERIFICATION
Design is complete. Test is being executed or analysis is underway.
VERIFIED
Verification complete with accepted evidence. Requirement is closed.
WAIVED
Requirement formally waived with documented rationale and authority sign-off.
CLOSED
Deleted or superseded. Archived with reference to replacement requirement.

Change Control Workflow

This is the step-by-step process a requirement change follows in Notion, from discovery to implementation. Every step has a specific Notion action.

1
Engineer identifies need for change
During design, test, or customer discussion, an engineer discovers a requirement needs to change. They open the requirement page in Notion and add a comment flagging the issue with supporting data.
Notion: Add comment on req page
2
Submit a Change Request
Create a new entry in the Change Requests database. Fill in the CR template fully — problem statement, proposed change, impact analysis, and class determination. Incomplete CRs are returned without review.
Notion: New CR entryStatus → SubmittedLink affected reqs
3
Mark affected requirements "In Change"
Update the Status of every requirement listed in the CR's "Affected Requirements" relation to "In Change." This signals to the team that the design basis is under review and implementation should pause on those reqs.
All affected reqs: Status → In Change
4
Review board assessment
Class I: Program Lead convenes a review (can be async in Notion comments). Class II: Subsystem Lead reviews solo. Both use the CR page to document the impact analysis review. A decision must be made within the SLA (Class I: 5 days, Class II: 2 days).
Status → Under ReviewNotion: comments for review discussion
5
Decision recorded
Decision authority fills in the Disposition field of the CR and updates Status to Approved or Rejected. Both decisions require written rationale — approvals explain why the change was necessary; rejections explain what alternative path will be pursued.
CR Status → Approved or RejectedDisposition field required
6
If Approved — implement the change
Update the requirement text in the database. Add a row to the Change History toggle in the req page. If "V&V Re-run Required" was checked on the CR, mark the linked V&V Records as "Retest Required." Update any downstream T3/T4 requirements that are now inconsistent. Return all affected req statuses to Baselined.
Update req textReq Status → BaselinedFlag V&V retestsCR Status → Implemented
7
Close the CR
Confirm all checklist items on the CR page are complete. Update CR Status → Closed. The CR remains permanently in the database as an audit trail — never delete CRs.
CR Status → ClosedNever delete — archive trail

V&V Traceability Workflow

The V&V flow answers the question: "Which requirements have been closed, and with what evidence?" The Notion relation between T2/T3 and V&V Records is your compliance matrix.

Key Concept

One V&V Record can verify multiple requirements. A bench test that sweeps temperature, voltage, and load simultaneously can close a dozen T3 reqs in a single test event. Conversely, one requirement may require multiple verification events (e.g., test at ambient + test at temperature extremes). Model this by linking the V&V Record to all reqs it closes — requirements can have multiple linked V&V Records.

The V&V Plan (Built at PDR)

Before the first prototype is built, create placeholder V&V Records for every planned test event. Set Result = "Pending" and link the requirements they will verify. This gives you a live compliance matrix that starts "all red" and goes green as tests are executed.

Planned V&V Record — Before Test
VV ID: VV-014 Title: Laser driver peak current — Proto-1 bench test Method: Test Phase: Proto Reqs Linked: LASER-003, LASER-004, LASER-009, PWR-017 Result: Pending ← placeholder until executed Date: ← blank until executed Procedure: https://drive.google.com/[link] Evidence: ← blank until executed

Closing a Requirement

  1. Execute the test. Collect and store data in a test report (Google Drive, Notion file upload, or your engineering file server).
  2. Open the V&V Record in Notion. Fill in Date, Performed By, Build/Unit, Evidence link.
  3. Set Result to Pass, Fail, or Conditional Pass.
  4. For each linked requirement: if the result is Pass, update V&V Status on the T2/T3 requirement to "Verified."
  5. For Fails: open a separate issue or action item. Do not close the V&V Record — update Result = Fail and leave evidence linked.

Generating a Compliance Matrix

Create a linked view of the T2 or T3 database filtered and grouped by V&V Status. This is your compliance matrix — it answers "what percentage of requirements are verified" at any point in the program.

Compliance Matrix View — Notion Filter Config

Naming Conventions

Consistent IDs are non-negotiable. Every requirement, CR, and V&V record needs a stable, human-readable identifier that appears in design documents, test reports, and emails — not just in Notion.

ID Format Reference
// Tier 1 — Mission Requirements MR-001 MR-002 MR-003 ... // Tier 2 — System Requirements SR-001 SR-002 SR-003 ... // Tier 3 — Subsystem Requirements (prefix = subsystem) LASER-001 Laser transmitter subsystem RX-001 Receiver / detector subsystem TDC-001 Time-to-digital converter subsystem SCAN-001 Scanning mechanism GEO-001 Georeferencing (IMU + GNSS) PWR-001 Power management DATA-001 Data interface // Tier 4 — Implementation Requirements MFG-001 Manufacturing ASSY-001 Assembly QA-001 Quality / inspection // Change Requests CR-001 CR-002 ... // Verification & Validation Records VV-001 VV-002 ... // Rules: // — IDs are assigned sequentially and NEVER reused // — Closed/superseded reqs keep their ID forever // — ID format must match in Notion AND in design docs // — Use 3-digit minimum padding: 001, not 1

Weekly Operations & Review Cadence

Daily (5 minutes)

Each engineer checks "My Requirements" view. Looks for: status changes needed, new comments requiring response, V&V records to fill in from yesterday's tests.

Confluence / Wiki Alternative

If your team uses Confluence instead of Notion, the architecture is the same but the implementation differs. Confluence lacks native relational databases, so you implement traceability differently.

Tradeoff

Confluence pages are better for rich documentation but lack Notion's relational database power. For requirements management, Confluence works best when paired with Jira — Confluence holds the narrative, Jira holds the requirements as issues with links and filters.

Confluence Requirement Page Structure

Each T2/T3 requirement is a row in a Confluence table, with a linked Confluence page for detail. Use the "Requirements Blueprint" Confluence template if available, or create a custom blueprint. Each detail page mirrors the Notion template structure above.

For Pure Wiki (no Jira)

If you're truly wiki-only (GitHub Wiki, Notion without databases, Obsidian), the minimum viable system is:

  • One markdown file per requirement, named by ID (e.g. LASER-003.md).
  • A master index table in a README.md or REQUIREMENTS.md file with status and verification columns.
  • Git commit discipline: every requirement change is a commit with the CR ID in the message (git commit -m "CR-014: Update LASER-003 peak current spec to 30A").
  • Change history lives in git log --follow LASER-003.md — permanent and auditable.
  • This approach scales to ~200 requirements before becoming unwieldy. Past that, move to a proper relational system.
Recommendation

For a hardware startup doing serious prototype-to-production work, Notion is the right tool up to ~500 T3 requirements. Past that scale, or when regulatory compliance demands (DO-178, AS9100, ISO 13485), migrate to Jama Connect or Polarion which are purpose-built for requirements traceability and can generate compliance matrices automatically.
